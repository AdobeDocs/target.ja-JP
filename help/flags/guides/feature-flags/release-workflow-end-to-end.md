# エンドツーエンドのリリースワークフロー {#release-workflow}

このページでは、リリースマネージャーが管理する調整リリースに関するアクティビティの全順序について説明します。

## &#x200B;1. アプリケーションごとに機能フラグを定義 {#define-flags}

各製品チームには、**製品リリース所有者**&#x200B;が割り当てられ、フラグコンソールにログインして、所有するアプリケーションの機能フラグを作成します。 次に、製品チームはコードに条件付きロジックを実装し、これらのフラグの背後に機能を配置します。

## &#x200B;2. リリースの作成 {#create-release}

新しいリリースを作成するには、サポートリクエストが必要です。完全にセルフサービスではありません。 フラグのサポートに連絡して、リリースを作成します。 リリース名、所有チーム、ターゲット環境、目的、参加アプリケーション、および想定される期間を指定します。

リリースが確認されると、リリースマネージャーはコンソールを開き、リリース設定を完了します。

## &#x200B;3. リリースに機能フラグを追加する {#add-flags}

リリースに追加されたアプリケーションごとに、アプリケーションの製品リリース所有者がログインし、関連する機能フラグをリリースに追加します。 これらのフラグは、リリースで公開される機能を表します。

## &#x200B;4. オーディエンスの設定 {#configure-audience}

リリースマネージャーは、リリースのオーディエンス（例：特定の国のユーザーの1%、特定のメールドメインを持つすべてのユーザー、特定のユーザーIDのリスト）を選択します。 設定が完了すると、リリースマネージャーはリリースを保存して公開し、指定したオーディエンスに対して公開します。

## &#x200B;5. ロールアウトの展開と管理 {#expand}

リリースの公開後、リリースマネージャーはオーディエンスルールを調整して、ロールアウトを徐々に展開し、問題を監視し、リリース状態コントロールを使用してライフサイクルを管理できます。 詳しくは、[ リリースの状態](release-states.md)を参照してください。

## キーポイント {#key-points}

* すべての参加アプリケーションは、機能フラグの後ろに独立してコードを本番環境にデプロイします。調整されたデプロイメントのタイミングは必要ありません。
* このリリースは、チーム間ですべてのフラグを同時にアクティブ化する単一の制御ポイントです。
* リリースのオーディエンスターゲティングは、認証トークン属性（国、電子メール、ユーザーID、割合）にもとづいて行われます。
