# ホスト

サイトと実稼動前の環境を整理して、[!DNL Adobe Target]での管理と個別レポートを容易にします。

ホスト管理の主な目的は、非アクティブなコンテンツが誤って Web サイトに表示されるのを確実に防ぐことです。ホスト管理では、[環境](/help/administrating-target/environments.md)別にレポートデータを分割することもできます。

ホストとは、[!DNL Target]要求が行われたドメインのことです。 Webサイトでは、通常、[!DNL Target]リクエストをおこなうURLの`location.hostname`プロパティです。

デフォルトでは、[!DNL Target]は[!DNL Target]リクエストをおこない、[!DNL Target]応答を受け取るホストを制限しません。 新しいホストがリクエストを行うと、自動的に機能します。 また、不明なドメインや予測できないドメインでのテストも可能です。 このデフォルトの動作を上書きする場合は、[!DNL Target]で動作するホ許可リストストを制限するまたブロックリストはホストを設定できます。

ホストを管理するには、**[!UICONTROL 管理]** /**[!UICONTROL ホスト]**&#x200B;をクリックします。

![](assets/hosts_list.png)

## ホストの認識 {#concept_0D4B43E23AA9408F8B28A57ED754BF65}

ホストを認識し、[!UICONTROL ホスト]リストに追加するには、次の条件を満たす必要があります。

* 少なくとも1つの[!DNL Target]要求がホストに存在する必要があります
* ホスト上のページには、以下のものが含まれている必要があります。

   * 正確なat.jsの参照
   * [!DNL Target]リクエストまたは自動生成されたグローバル[!DNL Target]リクエスト

* [!DNL Target]リクエストを含むページは、ブラウザーで表示する必要があります

ページが表示されると、ホストが[!UICONTROL ホスト]リストに表示され、環境でホストを管理したり、アクティビティおよびテストをプレビューおよび開始したりできます。

>[!NOTE]
>
>この説明は、個人で使用する開発サーバーも対象となります。

ホストが[!UICONTROL ホスト]リストに追加された後、確実にホストが認識されるようにします。

1. **[!UICONTROL 管理]** / **[!UICONTROL ホスト]**&#x200B;をクリックします。
1. ホストが一覧に表示されていない場合は、ブラウザーを更新します。

   デフォルトでは、新しく認識されたホストは[!UICONTROL 実稼動]環境に配置されます。 [!UICONTROL 実稼動]環境は最も安全な環境です。これは、これらのホストから非アクティブなアクティビティを表示できないからです。

1. （条件付き）**[!UICONTROL 移動]**&#x200B;アイコン（ ![移動アイコン](/help/administrating-target/assets/icon-move.png) ）をクリックして、ホストを[!UICONTROL 開発]、[!UICONTROL ステージング]または他の環境に移動します。

>[!NOTE]
>
>[!UICONTROL 実稼動]環境は、名前を変更した場合でも削除できません。 この環境で、最終的なアクティブなアクティビティとテストを扱うことを前提としています。 デフォルトの環境では、非アクティブなキャンペーンの表示は許可されていません。

## ホストのリストの並べ替えまたは検索 {#section_068B23C9D8224EB78BC3B7C8580251B0}

「[!UICONTROL ホスト]」リストを並べ替えるには、任意の列ヘッダー（[!UICONTROL 名前]、[!UICONTROL 環境]または[!UICONTROL 最後に要求された]）をクリックして、リストを昇順または降順に並べ替えます。

「[!UICONTROL ホスト]」リストを検索するには、「[!UICONTROL ホストを検索]」ボックスに検索語句を入力します。

## &lt;a0許可リスト/>要求を[!DNL Target]に送信する権限のあるホストを指定するを作成します。{#allowlist}[!DNL Target]

[!DNL Target]要求を[!DNL Target]に送信する権限を持つホスト（ドメイン）を指定する許可リストを作成できます。 リクエストを生成するその他のすべてのホストに対しては、コメントアウト認証エラーの応答が返されます。 デフォルトでは、[!DNL Target]リクエストを含むホストは、[!UICONTROL 実稼動]環境の[!DNL Target]に登録され、アクティブで承認済みのすべてのアクティビティへのアクセス権を持ちます。 この方法が望ましくない場合は、代わりにを使用し許可リストて、[!DNL Target]リクエストをおこない、[!DNL Target]コンテンツを受け取る資格のある特定のホストを記録できます。 すべてのホストは引き続き[!UICONTROL ホスト]リストに表示され、環境を使用してこれらのホストをグループ化し、ホストがアクティブなアクティビティや非アクティブなアクティビティを表示できるかどうかなど、異なるレベルを各ホストに割り当てることができます。

次の手順で許可リストを作成します。

1. 「[!UICONTROL ホスト]」リストで、「**[!UICONTROL ホストを承認]**」をクリックします。
1. 「**[!UICONTROL コンテンツ配信用に認証済みホストを有効にする]**」切り替えを有効にします。
1. 必要に応じて、「**[!UICONTROL ホストに]**&#x200B;を含む」ボックスに目的のホストを追加します。

   複数のホストを追加することもできます。この場合、1 行に 1 つずつ表示されます。

1. 必要に応じて、「**[!UICONTROL ホストに]**&#x200B;を含まない」ボックスに目的のホストを追加します。

   複数のホストを追加することもできます。この場合、1 行に 1 つずつ表示されます。

1. 「**[!UICONTROL 保存]**」をクリックします。

権限のないホストで[!DNL Target]リクエストがおこなわれると、呼び出しは`/* no display - unauthorized mbox host */`で応答します。

>[!IMPORTANT]
>
>**セキュリティのベストプラクティス**:のubox機能を使用する場合、この [!DNL Target]で許可リストは、リダイレクト先のドメインのリストも制御 [](/help/c-implementing-target/c-non-javascript-based-implementation/working-with-redirectors.md) されます。uboxを実装の一部として使用する場合は、リダイレクト先のドメインを必ず追加してください。 が指定されていない場許可リスト合、[!DNL Adobe]はリダイレクトURLを検証できず、悪意のあるリダイレクトから保護できません。
>
>許可リストは、環境よりも優先されます。 ホスト機能を使用する前に許可リストすべてのホストを消去し、ホストで許可されているホ許可リストストのみをホストリストに表示します。 その後、ホストを必要な環境に移動できます。

場合によっては、他のサイトからのドメインが環境内に表示されることがあります。ドメインがat.jsを呼び出すと、そのドメインがリストに表示されます。 例えば、自分の Web ページのいずれかが他のユーザーのサーバーにコピーされた場合は、そのサーバーのドメインが自分の環境に表示されます。スパイダーエンジン、言語翻訳サイトまたはローカルディスクドライブからのドメインが表示されることもあります。

`mboxHost` が API 呼び出しで渡される場合、渡される環境のコンバージョンが記録されます。環境が渡されない場合、この呼び出しのホストはデフォルトで[!UICONTROL Production]に設定されます。

また、「[!UICONTROL ホストに次を含まない]」ボックスに目的のホストを追加することブロックリストで、[!DNL Target]要求を[!DNL Target]に送信できないホスト（ドメイン）を指定するを作成することもできます。

>[!NOTE]
>
>[!UICONTROL Authorized Hosts]リストは、[!DNL Target]ホストとデフォルトのリダイレクトホストの両方に使用されます。 [!DNL Adobe Target] JavaScript SDK(at.js)*AND*&#x200B;を使用することを承認した既存のドメインを、uboxのデフォルトのリダイレクトURLで使用するすべてのドメインに追加します。 今後、類似した新しいドメインを許可リストに追加します。

## ホストの削除 {#section_F56355BA4BC54B078A1A8179BC954632}

不要になったホストは削除することができます。

1. 「[!UICONTROL ホスト]」リストで、「****&#x200B;を削除」アイコンをクリックします。
1. 「**[!UICONTROL 削除]**」をクリックして削除を確定します。

>[!NOTE]
>
>ホスト上の[!DNL Target]リクエストを含むページを参照すると、そのホストが再び表示されます。

## ホストのトラブルシューティング {#concept_B3D7583FA4BB480382CC7453529FE1B7}

ホストにおいて問題が発生した場合は次のトラブルシューティングのヒントを試してください。

**お使いのアカウントのリストにホストが表示されません。**

* ブラウザーで[!UICONTROL ホスト]ページを再読み込みします。
* at.js参照を含め、[!DNL Target]リクエストが正しいことを確認します。
* ホスト上の[!DNL Target]リクエストの1つを参照してみます。 ホスト上の[!DNL Target]リクエストがブラウザーでレンダリングされなかった可能性があります。

**ランダムなドメインまたは不明なドメインが[!UICONTROL ホスト]リストに表示される。**

[!DNL Target]への要求がドメインからおこなわれると、このリストにドメインが表示されます。 場合によっては、スパイダーエンジン、言語翻訳サイトまたはローカルディスクドライブからのドメインが表示されることがあります。表示されるドメインがチームで使用していないドメインである場合は、「[!UICONTROL 削除]」をクリックして、そのドメインを削除します。

**リクエスト [!DNL Target] が/* no display - unauthorized mbox host */を返します。**

[!DNL Target]リクエストが権限のないホストでおこなわれた場合、リクエストは/* no display - unauthorized mbox host */で応答します。
