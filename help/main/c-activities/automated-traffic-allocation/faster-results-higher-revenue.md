# [!UICONTROL 自動割り当て]を使用すると、手動テストよりも迅速なテスト結果と高い収益が得られます

手作業のA/B アクティビティでは、アクティビティが完了するまでオーディエンス全体に優れたエクスペリエンスを提供できないため、コンバージョンが失われる可能性があります。 トラフィックの配分は、いくつかのエクスペリエンスが他のエクスペリエンスよりも優れていることを認識した後も固定されたままになり、勝者に対して行動を起こすには、アクティビティ全体を実行する必要があります。

## トラフィックの自動割り当て

サンプルサイズ、信頼性レベル、およびその他の統計的概念を選択するセットアップと計算コストを同時に削除または削減しながら、アクティビティの勝者エクスペリエンスをより頻繁かつ早期に提供するオプションが必要な場合は、[!UICONTROL 自動配分]が最適なオプションです。

## [!UICONTROL 自動割り当て]の仕組み

[!UICONTROL 自動割り当て]は、マルチアームドバンディットの原則を使用します。 この用語がなじみのない場合、ワンアームドバンディットはスロットマシンの口語の用語です（考える：ラスベガス）。 トラフィックの自動割り当てを、複数のスロットマシン（この場合はバリエーションをテストし、最初にすべてのハンドルを同じように引き出す）を持つように視覚化します。 時間の経過とともに、1つまたは複数のマシン、またはテストのバリエーションが、他のマシンよりも多く支払われる場合があります。 この状況が起こると、ギャンブラーは自然に勝つ人のハンドルを引っ張り始めます。 トラフィック配分の観点から、[!DNL Target]はより多くの訪問者に、より多くの成果を上げているエクスペリエンスまたはエクスペリエンスを提供します。

次の2週間のA/B アクティビティの例を考えてみましょう。 [!UICONTROL 自動割り当て]では、勝者エクスペリエンスが現れるにつれて、[!UICONTROL Target]はテストの早い段階で、より多くのトラフィックをその勝者に振り分けます。

![自動配分の図](/help/main/c-activities/automated-traffic-allocation/assets/Auto-Allocate-test.png)

## [!UICONTROL 自動割り当て]では、より高速な結果を得るにはどうすればよいですか？

メリットは明らかです。より多くの訪問者が、パフォーマンスが最も高いバリエーションを目にすることができます。 また、ひとつのバリエーションが完成するにつれ、テストの実行中にさらに多くの訪問者が勝者エクスペリエンスに振り向けられます。 この方法は、実行中のA/B アクティビティが、ホリデーシーズン、製品リリース、ワールドニュースイベントなどのコアビジネスの瞬間に発生している場合に特に役立ちます。

## [!UICONTROL 自動割り当て]で高い収益を得るにはどうすればよいですか？

[!UICONTROL 自動配分]は、手動のA/B分割よりも速く勝者を見つけ出します。また、従来のアプローチや手動アプローチで失われた収益を即座に獲得し、その勝者を活用することもできます。 [!UICONTROL 自動割り当て]は、最もコンバージョン率の高いエクスペリエンスに多くのトラフィックを割り当てるため、アクティビティの実行中および学習中に収益が増加する可能性があります。

次の例では、[!UICONTROL 自動割り当て]は、最も高いコンバージョン率を示したエクスペリエンス Dにより多くのトラフィック（40%）をプッシュすることで、テスト中により多くの収益を得ました。

![自動割り当てにより、より高い収益が得られます](/help/main/c-activities/automated-traffic-allocation/assets/five-experiences.png)

## どのような場合に手作業によるトラフィック配分を継続すればよいですか？

それぞれのエクスペリエンスのパフォーマンスを比較して優先順位を付ける必要がある場合は、手動のA/B テストが最も適しています。 [!UICONTROL 自動割り当て]は、パフォーマンスの高いユーザーを検索して活用しますが、パフォーマンスの低いエクスペリエンス間での差別化は保証されません。 手動でのトラフィック配分を利用することで、訪問者のトラフィックがどの程度テストの各バリエーションに含まれるかを完全に制御し、自社のビジネスに関連する統計的しきい値をカスタマイズすることができます。
